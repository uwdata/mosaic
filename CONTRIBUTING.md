# Contributing to Mosaic

Thanks for your interest in contributing to Mosaic! This guide covers how to set up a local development environment, run the test suite, and work on the documentation. For an overview of the repository's packages, see the [Repository Structure](README.md#repository-structure) section of the README.

Bug reports, feature requests, and questions are welcome via [GitHub issues](https://github.com/uwdata/mosaic/issues). For larger changes, please open an issue to discuss the design before sending a pull request.

## Prerequisites

- [`pnpm`](https://pnpm.io/) and [Node.js](https://nodejs.org/) version 18 or higher.
- [`uv`](https://docs.astral.sh/uv/) for the Python packages (`duckdb-server` and the Jupyter `widget`).
- A Rust toolchain and/or Go are only needed if you work on `duckdb-server-rust` or `duckdb-server-go`.

## Getting started

* Clone [https://github.com/uwdata/mosaic](https://github.com/uwdata/mosaic).
* Run `pnpm i` to install dependencies.
* Run `pnpm build` to build the client-side bundles.
* Run `pnpm test` to run the test suite.
* Run `uv build --all-packages` to build the Python packages.

## Running examples locally

* Run `pnpm dev` to launch a local web server and view examples. By default, the examples use DuckDB-WASM in the browser. We recommend using Firefox since it remembers the selected dropdown across browser reloads. For greater performance, launch and connect to a local DuckDB server as described below.

To launch a local DuckDB server:

* Install [uv](https://docs.astral.sh/uv/), if not already present.
* Run `pnpm server` to launch the [`duckdb-server`](https://github.com/uwdata/mosaic/tree/main/packages/server/duckdb-server). This runs the server in development mode, so the server restarts if you change its code.

## Documentation

The documentation site is built with [VitePress](https://vitepress.dev/) from the `docs/` directory.

* `pnpm docs:dev` — start the docs dev server with hot reload.
* `pnpm docs:build` — build the static site into `docs/.vitepress/dist`.
* `pnpm docs:preview` — serve the built site locally (run `docs:build` first).
* `pnpm docs:examples` — regenerate the example specs and their outputs (see below).

### LLM-friendly docs (`llms.txt`)

The site publishes [`llms.txt`](https://llmstxt.org/) and `llms-full.txt` files for use with AI assistants. These are generated by [`vitepress-plugin-llms`](https://github.com/okineadev/vitepress-plugin-llms) (configured in `docs/.vitepress/config.js`) **only during `pnpm docs:build`** — they are not produced by the dev server.

As a result:

* On the **dev server** (`pnpm docs:dev`), requesting `/mosaic/llms.txt` returns a **404**. This is expected.
* To view them locally, run `pnpm docs:build` followed by `pnpm docs:preview`, then visit `/mosaic/llms.txt` and `/mosaic/llms-full.txt`.
* In production they are deployed at `https://idl.uw.edu/mosaic/llms.txt` and `https://idl.uw.edu/mosaic/llms-full.txt`.

If you add, remove, or rename docs pages, the files regenerate automatically on the next build — no manual edits needed.

### Example specifications

The example specs under `specs/` and `docs/public/specs` are generated, not hand-edited. After changing an example, run `pnpm docs:examples` to regenerate them. CI verifies that these generated outputs are in sync, so commit the regenerated files.

## Before opening a pull request

Please make sure the relevant checks pass locally before submitting. These mirror what runs in CI.

For the JavaScript/TypeScript packages:

```sh
pnpm run lint
pnpm run build
pnpm run test
pnpm run docs:examples   # then commit any regenerated specs
```

For the Python packages (`duckdb-server`, `widget`):

```sh
uv run ruff check
uv run ruff format --check
```

For `duckdb-server-rust` (only if you changed it):

```sh
cargo fmt --manifest-path=./packages/server/duckdb-server-rust/Cargo.toml -- --check
cargo clippy --manifest-path=./packages/server/duckdb-server-rust/Cargo.toml --all -- -D warnings
cargo test --manifest-path=./packages/server/duckdb-server-rust/Cargo.toml
```

For `duckdb-server-go` (only if you changed it):

```sh
cd packages/server/duckdb-server-go
go build -tags=duckdb_arrow ./...
go test -tags=duckdb_arrow ./...
```
